> Мы работаем над инструментом **marketing-monitor** — веб-сервис для ежемесячного мониторинга конкурентов CheckOffice.ru.
>
> **Стек:** Python, Flask + Waitress (WSGI), Gunicorn на Render.com (бесплатный tier). Фронтенд — один HTML-файл (`static/index.html`). Никаких фреймворков, чистый JS.
>
> **Деплой:** Render.com, репозиторий на GitHub. Чтобы изменения попали на сайт — нужно запустить в Терминале:
> ```
> cd ~/Desktop/Claude/Projects/CheckOffice/marketing-monitor
> git add -A && git commit -m "описание" && git push
> ```
> Render деплоит автоматически за 2–3 минуты.
>
> **Живая ссылка:** https://marketing-monitor.onrender.com
>
> **Что умеет сервис:** собирает данные по 6 конкурентам (сайты, sitemap, соцсети VK/Telegram), генерирует HTML/PDF отчёт. Сбор занимает 1.5–3 минуты, работает в фоновом потоке.
>
> **Архитектура фоновых задач:** POST /api/run → запускает поток → фронтенд каждые 4 сек опрашивает GET /api/job/current → статус задачи хранится в памяти (JOBS dict) И в файле data/current_job.json (устойчиво к перезапускам).
>
> **Конкуренты в конфиге:** CheckOffice, MD Audit, Service Inspector, Ритейлика, Мерасофт + ещё один. Конфиг COMPETITORS в начале app.py.
>
> **Ключевые файлы:**
> - `app.py` — весь бэкенд (3149 строк)
> - `static/index.html` — весь фронтенд
> - `data/current_job.json` — статус текущей задачи
> - `output/reports/` — готовые отчёты
> - `Procfile` — gunicorn команда для Render
> - `requirements.txt` — flask, waitress, reportlab

## НАЙДЕННЫЕ ОШИБКИ И ПРОБЛЕМЫ

### Критические (влияют на работу)

**1. lru_cache не сбрасывается между запусками**
`parse_sitemap` закэширована — данные sitemap из прошлого запуска используются в следующем. Если сайт конкурента обновился, мы этого не увидим до перезапуска сервера.

**2. Race condition в JOBS dict**
JOBS читается и пишется из разных потоков без блокировок. При двух одновременных запросах возможна потеря данных.

**3. Один job_id "current" на всех**
Если два пользователя нажмут кнопку одновременно — второй запуск сотрёт статус первого.

**4. Daemon thread без fallback**
Если Gunicorn worker перезапустится во время сбора — задача обрывается, но в current_job.json зависнет статус "running" до следующего рестарта сервера (сейчас мы это обрабатываем, но не всегда).

### Средние (влияют на качество данных)

**5. collect_web_mentions — последовательный**
7 внешних сайтов опрашиваются по очереди. Можно параллельно — ускорит на ~30 сек.

**6. Нет глобального таймаута на весь сбор**
Каждый запрос ограничен 7–12 сек, но если зависнет много запросов подряд — сбор может идти 10+ минут.

**7. request_text не проверяет Content-Type**
Если сайт конкурента вернёт PDF или картинку вместо HTML — функция попытается декодировать бинарник как текст.

**8. JOB_FILE хранит весь отчёт**
`data/current_job.json` содержит полные данные (data: asdict_data) — несколько МБ. При каждом опросе это читается целиком.

### Мелкие (код и надёжность)

**9. 42 широких except-блока**
`except Exception: pass` в 42 местах. Реальные ошибки скрываются, сложно дебажить.

**10. lru_cache с timeout аргументом**
`parse_sitemap(site_url, timeout=10)` — lru_cache кэширует по обоим аргументам. Если где-то вызвать с другим timeout — это будет отдельная запись кэша.
